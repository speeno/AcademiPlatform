# 프롬프트 쿡북 (개발자용) — 에이전트 코딩 공용

> 이미 코드를 아는 사람이 **하네스(Claude Code / Cursor)** 로 생산성을 높이는 패턴 모음. 복붙해서 쓰세요.

---

## 1. 스펙 → 플랜 → 실행 (Plan-first)
큰 작업은 바로 시키지 말고 **계획부터** 받는다.
```
이 변경의 목표: (한 문장). 영향 받는 파일과 단계별 계획을 먼저 제시해줘. 코드는 아직 쓰지 말고.
```
→ 계획을 검토·수정한 뒤 "이 계획대로 진행해줘".
> Claude Code는 **플랜 모드**, Cursor는 계획을 먼저 요청하는 방식으로 활용.

## 2. 맥락(컨텍스트) 제대로 주기
- 관련 파일을 **열어두거나 명시**한다: `@파일경로` / "다음 파일을 참고: ..."
- 프로젝트 규칙은 **`CLAUDE.md`**(또는 Cursor Rules)에 적어두면 매번 자동 반영(코딩 컨벤션, 테스트 방법, 금지사항).
- 재현 정보: 오류 로그/스택트레이스/재현 절차를 **그대로** 붙인다.

## 3. 작게, 검증 가능하게
```
변경을 작은 단위로 나눠서 진행하고, 각 단계 후 어떻게 검증할지 알려줘.
```
- 변경(diff)은 **반드시 읽고 수락**. 한 번에 거대한 변경이면 "더 작게 쪼개줘".

## 4. 버그 수정
```
증상: (재현 절차/입력). 기대: (정상 동작). 실제: (오류/스택트레이스 붙여넣기).
원인 가설을 1~2개 제시하고, 가장 가능성 높은 것부터 최소 수정으로 고쳐줘. 회귀 테스트도 추가해줘.
```

## 5. 테스트 작성
```
이 함수/모듈의 동작을 표로 정리하고, 경계·예외 케이스를 포함한 단위 테스트를 작성해줘. (테스트 프레임워크: ___)
```

## 6. 리팩터
```
동작을 바꾸지 말고(테스트로 보장), 가독성/중복만 개선해줘. 변경 전후로 테스트가 통과하는지 확인해줘.
```

## 7. 코드 리뷰 (에이전트에게)
```
이 diff를 리뷰해줘. 버그·엣지케이스·보안·성능 관점에서 확신도와 함께 지적하고, 사소한 스타일은 제외해줘.
```
> 발견을 "모두 보고하게" 하고, 중요도/확신도 필터는 사람이 한다.

## 8. 위임(서브에이전트) — 큰 작업
- 독립적인 하위 작업(여러 파일 조사, 광범위 검색)은 **서브에이전트에 위임**하면 메인 맥락을 아낀다(Claude Code의 에이전트/Explore).
- 패턴: "이 기능과 관련된 코드 위치를 먼저 조사해서 요약해줘" → 결과 받고 본 작업.

## 9. Claude Code 팁
- `CLAUDE.md`로 프로젝트 규칙·자주 쓰는 명령을 고정.
- 플랜 모드로 위험한 변경 전 계획 검토.
- 슬래시 커맨드/훅으로 반복 작업·검증 자동화(테스트·린트 자동 실행 등).

## 10. Cursor 팁
- 관련 파일을 컨텍스트에 추가(@), Rules로 규칙 고정, Agent 모드로 다단계 작업.
- 제안 diff를 **부분 수락**으로 통제.

## 11. 안전·검증 (개발 트랙)
- ❌ 비밀번호·토큰·`.env`·고객 데이터 붙여넣기 금지.
- ✅ AI 변경은 **테스트·실행·diff 리뷰**로 검증 후 머지.
- ✅ 생성 코드의 라이선스·보안(인젝션·비밀유출) 점검.
