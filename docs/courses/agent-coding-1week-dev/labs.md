## Day 4 — 오케스트레이션 · 컨텍스트 · 평가

### 랩 4-1 — 작업 분할 & 위임 (오케스트레이션)
하나의 큰 작업을 **조사 → 구현 → 검증**으로 나눠 진행:
```
[조사] 이 기능과 관련된 코드·테스트·영향 범위를 조사해서 요약만 해줘(수정 금지).
```
```
[구현] 위 조사 결과를 바탕으로 계획을 세우고, 합의 후 작은 단위로 구현해줘.
```
```
[검증] 방금 변경을 '리뷰어' 관점에서 교차 검증해줘: 버그·엣지케이스·테스트 누락을 확신도와 함께.
```
- 관찰: 단일 에이전트로 한 번에 시킨 경우와 **품질/맥락 관리** 차이. "언제 분할이 이득인가" 토론.
- (심화, 선택) 독립 하위작업을 **병렬 위임**해 시간 단축 — 도구별 방법은 공식 문서.

### 랩 4-2 — 컨텍스트 다이어트 (전/후 비교)
같은 작업을 두 가지 방식으로 시키고 비교:
- **A(과다)**: 레포 전체를 막연히 맡기고 지시.
- **B(최적)**: 관련 파일만 명시 + `CLAUDE.md` 규칙 + 불필요 맥락 제거.
- 비교 항목: 결과 정확도 / 수정 반복 횟수 / (가능하면) 토큰·시간.
- 교훈 정리: "무엇을 넣고 뺄지"가 비용·품질을 좌우. 검색·캐싱·압축은 공식 문서 기준으로 적용.

### 랩 4-3 — 미니 평가 세트
스킬/프롬프트의 **합격 기준**을 정해 측정:
1. `test-writer` 스킬을 샘플 함수 3개에 적용.
2. 합격 기준 정의: "생성 테스트가 모두 통과 + 경계/예외 케이스 ≥1 포함".
3. 3개 중 몇 개가 기준을 충족하는지 표로 기록 → 스킬을 다듬어 재측정.
> 핵심: "느낌"이 아니라 **기준·측정**으로 스킬 품질을 관리.

## Day 5 — 표준화 + 거버넌스

### 랩 5-1 — 표준화 스프린트 (레포에 커밋)
우리 레포에 아래를 실제로 적용·커밋:
- `.skills/` — 검증된 팀 스킬셋(2~4개)
- `CLAUDE.md` — 컨벤션·테스트·금지사항·자주 쓰는 명령
- 자동화 — `.github/workflows/`(테스트/CI), (선택) 변경 시 검증 훅
- 가드레일 — "위험 작업은 플랜·리뷰 필수" 등 규칙 문서화
- 검증: 테스트·CI 통과 + diff 리뷰 후 머지/PR.

### 랩 5-2 — 팀 에이전트 워크플로우 표준 문서
`docs/agent-workflow.md` 작성(예시 골자):
```markdown
# 팀 에이전트 워크플로우 표준
## 언제 어떻게 쓰나
- 플랜-퍼스트로 계획 검토 후 실행, 변경은 diff 리뷰 + 테스트 통과 후 머지.
## 스킬셋
- 사용 스킬 목록과 용도(.skills/...)
## 자동화
- PR CI(node --test), (선택) 자동 리뷰
## 안전·거버넌스
- 비밀정보 금지, 토큰 최소권한, 위험 작업 게이트
## 도입 지표
- 적용 PR 비율 / 리드타임 / 리뷰 지적 감소 등
```

### 데모 (팀당 7분)
- 무엇을 표준화/자동화했나 → 시연 → 효과·도입 계획. (라이브 실패 대비 녹화)
