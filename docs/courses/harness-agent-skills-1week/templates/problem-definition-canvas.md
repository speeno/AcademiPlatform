# 문제 정의 캔버스 (Problem Definition Canvas)

> 만들기 전에 "무엇을, 왜" 부터 합의합니다. 팀이 함께 칸을 채우세요.
> 작성 후 강사 피드백(게이트)을 받고 빌드를 시작합니다. 한 칸이라도 비면 아직 빌드하지 마세요.

---

**팀명**: ____________   **작성일**: ____________

### 1. 문제를 한 문장으로
> "누가" + "어떤 상황에서" + "무엇이 불편/비효율적이다"
- 예) 운영팀이 매주 정산 자료를 **여러 엑셀에서 수작업으로 합치느라** 2시간이 걸린다.
- 우리 문제: ________________________________________________

### 2. 누가 겪나 (대상 사용자)
- 주 사용자: ____________ / 얼마나 자주: ____________

### 3. 지금은 어떻게 하고 있나 (현재 방식)
- ________________________________________________

### 4. 왜 문제인가 (영향)
- 시간/비용/실수/스트레스 등 구체적으로: ____________________

### 5. 성공의 모습 (성공 기준 — 측정 가능하게)
> "이렇게 되면 성공" 을 숫자나 명확한 상태로.
- 예) 정산 자료 합치기가 **2시간 → 5분**, 수식 실수 0건.
- 우리 기준: ____________________________________________

### 6. 범위 (이번 주에 만들 MVP)
- **포함(이번에 만든다)**: ____________________________
- **제외(이번엔 안 한다)**: ____________________________
> 욕심내지 마세요. "가장 작은, 그래도 쓸모 있는" 한 가지에 집중.

### 7. 입력과 결과물
- **입력(무엇을 넣나)**: 예) 엑셀 파일 3개 / 텍스트 / 질문
- **결과물(무엇이 나오나)**: 예) 합쳐진 표 / 요약 문서 / 웹페이지 / 답변

### 8. 데이터 (있다면)
- 사용할 샘플 데이터: ____________  (※ 실데이터 금지, 가공본 사용)

### 9. 위험·제약
- 막힐 것 같은 점, 안전·개인정보 이슈: ____________________

---
### 강사 피드백 게이트 ✅
- [ ] 한 문장 문제가 명확한가
- [ ] 성공 기준이 측정 가능한가
- [ ] MVP 범위가 일주일 안에 가능한가
- [ ] 개인정보/기밀 이슈가 없는가
