## 강의 노트
### ① 문제 정의 & MVP
- 좋은 문제 = **한 문장**으로 "누가/언제/무엇이 불편" + **측정 가능한 성공 기준**.
- **MVP(최소 기능)**: 욕심내지 말 것. "가장 작은데도 쓸모 있는 한 가지." 나머지는 "제외"로 명시.
- 흔한 함정: 기능 욕심 → 일주일 안에 못 끝남. **범위를 줄이는 게 실력.**

### ② 기술·업무 분해
- 결과물을 **기능 단위**로 쪼갠다(F1, F2…). 각 기능은 **입력 → 처리 → 출력**으로 적는다.
- 필요한 **재료**(데이터/예시/연동)를 적는다.
- **태스크 보드**: 작은 단위로, "한 번의 바이브 코딩 시도"로 끝낼 크기. **1번 태스크는 항상 '가장 작은 성공'**.
- 분해표의 "AI에게 줄 한 줄 지시"가 곧 빌드 대본이 된다.

## 실습
### 랩 4-1 (50분) — 문제 정의 캔버스
- 팀이 캔버스 1~9번을 채운다. 특히 **성공 기준(측정 가능)** 과 **MVP 범위(포함/제외)** 에 시간 투자.

### 게이트 (30분) — 강사 승인
- 팀별로 강사가 4개 체크(명확성/측정가능/일주일 가능/안전) → **승인되면 빌드 시작**, 아니면 즉시 수정.

### 랩 4-2 (130분) — 분해 + 빌드 스프린트 1
- 분해표 완성 → 역할 분담(드라이버/내비/검증/기록).
- 빌드 시작: **1번 태스크(가장 작은 성공)** 부터. 예) "빈 페이지에 제목과 버튼 하나" → 동작 확인 → 다음.
- 작동 사이클: 지시 → 수락 → 실행/확인 → (오류면) 오류 붙여 수정 → 다음 태스크.
- **맥락 묶음**(분해표 E) 을 대화 시작에 붙여 품질 유지.

## 산출물
- **승인된 문제 정의 캔버스** + **기술·업무 분해표** + **빌드 중간본(최소 1~2개 기능 동작)**.

## 강사 팁 / 흔한 막힘
- **범위 과욕**이 1순위 위험 → 게이트에서 가차 없이 MVP로 축소.
- 빌드가 막힌 팀: "지금 되는 가장 작은 것부터 보여줘"로 되돌리고, 오류는 메시지째 AI에 전달.
- 한 명만 키보드 잡는 팀 → 역할 교대 강제(1시간 단위).
- 데이터 이슈: 실데이터 쓰려는 팀 → 즉시 가공/샘플로 전환.
- 진도 격차 큰 경우: 빠른 팀엔 "기능 1개 더" / 느린 팀엔 강사·조교 밀착.
