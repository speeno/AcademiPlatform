## Day 2 — 스킬 라이브러리 · 서브에이전트 · MCP

### 랩 2-1 — 팀 스킬 2~3개
`.skills/` 아래에 재사용 스킬 작성. 예시:

`.skills/pr-describer/SKILL.md`
```markdown
---
name: pr-describer
description: 변경(diff/커밋)을 받아 PR 설명을 작성할 때 사용. "PR 설명 써줘" 요청 시.
---
# PR 설명 작성 스킬
## 절차
1. 변경 요약(무엇을/왜)을 3~5줄로.
2. 변경 파일과 주요 변경점을 항목화.
3. 테스트 방법과 리스크/롤백을 적는다.
## 출력 양식
## 요약 / ## 변경점 / ## 테스트 / ## 리스크
## 규칙
- 사실만(코드에 없는 내용 금지). 한국어.
```
- 호출: `.skills/pr-describer/SKILL.md 절차로 이번 브랜치의 PR 설명을 써줘.`
- 같은 방식으로 `test-writer`(D1), `commit-message` 스킬도 추가.

### 랩 2-2 — 서브에이전트 위임
- 큰 작업 전에 **조사 위임**으로 메인 컨텍스트 절약:
```
이 기능과 관련된 코드 위치(파일·함수)를 먼저 조사해서 목록과 한 줄 설명으로 요약만 해줘. 수정은 하지 말고.
```
- 결과를 받고 본 작업 지시 → 위임 전/후의 컨텍스트·속도 차이 관찰.

### 랩 2-3 — MCP 연동 (예: GitHub)
- MCP로 외부 컨텍스트를 워크플로우에 연결(설정은 공식 문서 기준).
- 예시 활용:
```
연결된 GitHub에서 이 레포의 열린 이슈 중 "good first issue"를 가져와, 가장 작은 것 하나의 해결 계획을 세워줘.
```
- ⚠️ 토큰은 안전하게(화면 공유 주의), 최소 권한으로.

## Day 3 — 자동화 + 팀 적용

### 랩 3-1 — 변경 시 검증 자동화
- 목표: 코드가 바뀌면 **테스트·린트가 자동 실행**되도록.
- 방법(택1, 환경에 맞게):
  - **로컬 자동화**: 에디터/CLI 훅 또는 watch 스크립트로 저장·변경 시 `node --test` 실행.
  - **CI**: 간단한 GitHub Actions로 PR마다 테스트 실행.
- 예시 GitHub Actions (`.github/workflows/test.yml`):
```yaml
name: test
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: node --test
```
- AI에게: `이 레포에 PR마다 'node --test'를 돌리는 GitHub Actions를 추가해줘.`

### 랩 3-2 — 코드 리뷰 자동화(개념·실습)
```
이 PR(또는 diff)을 리뷰해줘. 버그·엣지케이스·보안 관점에서 확신도와 함께 지적하고, 사소한 스타일은 빼줘.
```
- (심화) MCP/CI와 결합해 PR에 자동 리뷰 코멘트를 남기는 흐름을 설계(도구별 방법은 공식 문서).

### 랩 3-3 — 팀 적용 스프린트
- 우리 레포에 **스킬셋(`.skills/`) + `CLAUDE.md` + 자동화(워크플로우)** 를 실제로 커밋.
- 데모: "무엇을 표준화/자동화했고, 효과는?"
